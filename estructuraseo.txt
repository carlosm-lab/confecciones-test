## Campos SEO a implementar

El usuario debe poder ingresar manualmente, por producto, los siguientes elementos. Para cada uno, durante la auditoría debes determinar si el proyecto ya tiene un mecanismo automático que lo genera, y resolver el conflicto antes de implementar (ver Fase 2):

- **Title** (`<title>` / meta title) — el título SEO específico de esa página de producto.
- **Description** (meta description) — la descripción que aparece en resultados de búsqueda.
- **Keywords** (meta keywords) — lista de palabras clave asociadas al producto.
- **X-Robots-Tag** — directivas de indexación (`index`/`noindex`, `follow`/`nofollow`, y combinaciones) específicas para esa página de producto.
- **Publisher** — el meta tag `publisher` (o el equivalente que el proyecto use, por ejemplo Open Graph `og:site_name` si así corresponde — verifícalo durante la auditoría, no asumas el nombre exacto del campo).
- **Campos adicionales relevantes de SEO por producto** que detectes como faltantes durante la auditoría y que sean coherentes con lo que el proyecto ya maneja a nivel global (por ejemplo: Open Graph title/description/image si el proyecto ya usa Open Graph en otras páginas, canonical URL si aplica, structured data específico del producto si el proyecto ya usa JSON-LD en otras páginas). **No agregues campos que el resto del proyecto no usa en ningún otro lugar** — mantén coherencia con lo que ya existe.

> Estos campos son **opcionales por producto**. Si el admin no los completa, el sistema debe seguir comportándose exactamente igual que hoy (usando lo que sea que genere automáticamente, o el valor por defecto que ya existe).

## Fase 3 — Diseño de la solución (sin escribir código todavía)

### 3.A — Modelo de datos

Define los campos nuevos a agregar a la tabla de productos en Supabase, todos **opcionales/nullable**, sin afectar ninguna columna existente:

```
COLUMNAS NUEVAS PROPUESTAS (todas nullable, sin default que altere productos existentes):
  seo_title           [tipo]
  seo_description     [tipo]
  seo_keywords        [tipo — decide si texto separado por comas o jsonb array, según
                        consistencia con cómo el proyecto ya maneja arrays en otros campos,
                        por ejemplo cómo están guardadas las "Etiquetas" actuales — reutiliza
                        ese mismo patrón si ya existe uno]
  seo_robots           [tipo — ej. texto, con valores válidos tipo "index,follow" /
                         "noindex,nofollow" / etc.]
  seo_publisher         [tipo]
  [cualquier campo adicional identificado como necesario en la Fase 1, Bloque C]
```

> Verifica el patrón de nombres de columnas ya existente en la tabla (¿usa `snake_case`? ¿tiene All ya un prefijo para campos de un grupo, como "Precios Especiales"?) y sigue exactamente esa convención para los campos nuevos.

### 3.B — Migración

Diseña la migración SQL como una operación puramente aditiva:

```sql
-- Ejemplo de estructura, ajustar nombres reales tras la Fase 1
alter table products
  add column if not exists seo_title text,
  add column if not exists seo_description text,
  add column if not exists seo_keywords text,
  add column if not exists seo_robots text,
  add column if not exists seo_publisher text;
```

- [ ] Confirma que `add column if not exists` con columnas nullable y sin default **no** dispara ningún trigger, RLS policy, o constraint que pueda romper inserts/updates existentes — verifica las policies de la tabla antes de aplicar la migración.
- [ ] Confirma que esta migración no requiere bloquear la tabla de forma que afecte producción si el proyecto está en uso activo (en una tabla con 100+ productos esto normalmente es instantáneo para un `add column` nullable, pero verifícalo).

### 3.C — Formulario (UI)

- Inserta la sección SEO **después** de la última sección existente del modal (la que sigue a "Producto Activo" según la captura de referencia, o donde la Fase 1 Bloque B haya determinado que termina el formulario actual — no reordenes nada existente).
- Usa el mismo patrón visual que la caja de "Precios Especiales" o "Datos del Catálogo Público" (encabezado con ícono, fondo diferenciado sutil, etiqueta "Opcional"), reutilizando las clases/estilos ya existentes en el proyecto para ese patrón — no crees una paleta o spacing nuevo.
- Cada campo nuevo debe tener su placeholder explicativo, igual que el resto de campos opcionales del formulario (compara con "Dejar vacío si no hay oferta" o "Dejar vacío si no aplica").
- El campo `X-Robots-Tag` debe presentarse de forma usable para un admin no técnico: idealmente checkboxes o un select con las combinaciones válidas (`index, follow` / `noindex, follow` / `index, nofollow` / `noindex, nofollow`) en lugar de un input de texto libre que el admin tenga que escribir a mano sin guía — pero confirma este patrón de UX contra cómo el proyecto ya resuelve selección de opciones en otros campos (por ejemplo cómo están implementadas las "Tallas Disponibles" como botones seleccionables) y reutiliza ese mismo patrón de componente si existe.

### 3.D — Guardado

- El payload que se envía a Supabase al crear/editar producto debe incluir los campos nuevos solo si tienen valor; si están vacíos, deben guardarse como `null`, nunca como string vacío, para que la Fase 2 (fallback automático) funcione de forma consistente con cómo el proyecto distingue "sin valor" en sus demás campos opcionales (confirma cómo lo hace hoy con "Precio Anterior" o "Precio Solo Mano de Obra", que ya son opcionales, y replica ese mismo patrón).

### 3.E — Consumo en la página pública

- Ajusta `generateMetadata` (o equivalente) en la página de detalle de producto para que: si el producto tiene `seo_title`/`seo_description`/etc., los use; si no, **se comporte exactamente igual que hoy** sin esos campos.
- Si el proyecto usa `X-Robots-Tag` como header HTTP (no como meta tag), implementa según la documentación oficial de Next.js para setear headers de forma condicional por ruta dinámica — no satures el `<head>` con un meta tag duplicado si el proyecto no usa ese patrón en otro lado.
